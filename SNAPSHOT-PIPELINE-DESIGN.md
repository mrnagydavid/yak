# Seed pipeline — snapshot redesign

> **✅ Status: implemented 2026-07-09** (shipped `version = sv-2026-06-01-86b7078c`). This is now both
> the migration record and the **living reference** for the current snapshot pipeline: the editing
> recipes (§7), checkers (§6/§6.1), and gotchas (§12) are current guidance; the ordered migration steps
> (§8) and the go/no-go gate are historical. Repo-root authority for how the seed is built and edited;
> `SPEC.md` §9 is the summary. (Originally written as an implementation plan — hence the future tense
> below.)

> Self-contained plan. Audience: a future Claude Code session that will implement it, plus a tech PM.
> Assumes **no** prior context beyond the repo. Read this top-to-bottom before touching anything.
>
> **Goal:** replace the layered seed pipeline with a **single, hand-editable snapshot** of the
> wordlist. To change a translation, add an example, or fix a part of speech, you edit one file and
> save. A build step minifies that file into what ships. The old layered machinery is **archived, not
> deleted** (git + a `legacy/` folder), so no expensive LLM work is lost and a future bulk re-pass is
> still possible.
>
> **Supersedes `SEED-PIPELINE-DESIGN.md`** (now archived at `data/seed/sv/legacy/SEED-PIPELINE-DESIGN.md`)
> for all day-to-day work. That doc describes the layered pipeline this plan retires; it is kept as the
> reference for the archived machinery.

## 1. TL;DR

- **Source of truth becomes `data/seed/sv/wordlist.json`** — a pretty-printed JSON array of the ~8,322
  shipped entries, one value per line, in today's order. You edit it by hand. Git is the history.
- **A new build step `seed:pack`** reads the snapshot, recomputes the mechanical/derived fields, and
  writes the shipped files (`seed-sv.json` + `version.json`) — exactly the files `seed:apply` writes
  today, so nothing downstream changes.
- **The migration is content-neutral.** Canonicalizing to one key order (§5.1) rehashes exactly **6**
  entries whose *content is unchanged* (only their key order differs in today's file), so the shipped
  `version` moves once from `sv-2026-06-01-4898e37e` to a precomputed **`sv-2026-06-01-86b7078c`**. The
  go/no-go gate: every `seedKey`'s decoded content is identical before/after (verified: 0 content
  diffs). Users re-download 6 byte-different-but-content-identical entries; seed-sync matches on
  `(seedKey, meaningKey)`, so no learner progress is affected.
- **A small set of checkers run in CI** (the GitHub Pages workflow runs `pnpm test` before it
  publishes). They read the finished wordlist and block a broken one. Most already exist and are
  reused; one is reworked; one is net-new.
- **The layered machinery is archived.** `base.json`, `layers.json`, `layers/`, and the
  batch/compile/stale scripts move to a `legacy/` area (still committed). They are the escape hatch for
  a future large-scale re-curation; they are not on the day-to-day path.

Everything we lose by dropping layers is either (a) scaffolding that only existed to protect against
problems the layers themselves created, or (b) a capability that never actually needed the layers
(see §9). Nothing that protects *users* is lost.

---

## 2. Why (one paragraph)

The layered pipeline was built for the **construction phase**: bulk LLM curation where the priority was
never losing expensive LLM output and knowing which of 8,000 words needed re-curation. That phase is
done. We are now in **maintenance**: occasional single-field edits. For that, layers are overhead — and
worse, they carry footguns (e.g. the reducer applies layer-40's translation *over* the "human override"
layer-90 for `translation`/`subDefinitions`, so the stated precedence lies; re-runs historically revived
stale answers). A single editable snapshot removes the ceremony and the footguns. Correctness moves from
"guaranteed by how the file was *built*" to "checked after the fact by dumb, reliable checkers at the CI
gate."

## 3. Decisions already made (do not relitigate)

| Decision | Choice |
|---|---|
| Snapshot format | **Pretty JSON** (2-space indent, one value per line), single file, stable order. Git is the sole revision history. |
| Minified shipped seed | **Keep committing it** (`data/seed/sv/seed-sv.json` + `public/seed-sv.json`), generated by `seed:pack`. A reproducibility test guards that the committed minified file equals what packing the snapshot produces. |
| Provenance ("why") | A human-written **changelog** (`data/seed/sv/CHANGELOG.md`) capturing the reasoning behind edits, on top of git blame. |
| Old machinery | **Archive, don't delete.** Move to `legacy/`, keep in git. |
| Safety anchor | **Content-neutral migration** (§5.1, Option A): every `seedKey`'s decoded content is unchanged; `version` moves once `4898e37e` → **`86b7078c`** (a content-neutral rehash of 6 key-reordered entries). |

## 4. Architecture: before → after

**Before** (layered):
```
data/seed/sv/
  base.json                 # immutable candidate list (8,425 rows)
  layers.json               # manifest: precedence + field ownership
  layers/                   # 10-cleaner 20-pos 30-subdef 40-translation 45-split 50-senses
                            #   60-examples 70-manual-examples 80-manual-flags 90-manual
                            #   (+ runs/ ledgers, decisions.json, stale.json)
  token-synonyms.json       # curated concept-merge verdicts (read by groupConcepts)
  seed-sv.json              # generated, committed, shipped
  version.json              # generated, committed
apply-decisions.mjs + lib/layers.mjs   # the reducer
```

**After** (snapshot):
```
data/seed/sv/
  wordlist.json             # ★ SOURCE OF TRUTH — pretty, hand-edited, committed
  token-synonyms.json       # kept — now a CHECKER input (concept detection), not a layer
  CHANGELOG.md              # ★ human "why" log
  seed-sv.json              # generated by seed:pack, committed, shipped   (unchanged filename)
  version.json              # generated by seed:pack, committed           (unchanged filename)
  legacy/                   # ★ archived: base.json, layers.json, layers/  (still in git)
scripts/seed/
  pack.mjs                  # ★ the new build (replaces apply-decisions on the day-to-day path)
  audit-*.mjs               # kept; audit-gloss reworked (§6)
  lib/layers.mjs            # kept for groupConcepts/normTr + the archived path
  legacy/                   # ★ archived batchers/compile/stale (optional relocation)
public/
  seed-sv.json  version.json   # generated by seed:pack, committed (as today)
```

### Authored vs derived

The snapshot contains **only authored fields**. `seed:pack` recomputes the **derived fields** so they
can never go stale:

- **Authored** (live in `wordlist.json`): `seedKey`, `lemma`, `pos`, `cefr`, `gender`, `ipa`,
  `inflections`, `subDefinitions`, `altMeanings` (incl. their `senseKey`, `gloss`, `examples`),
  `enUncountable`, `enProper`, `svUncountable`, `examples`, `translation`, `sense` (`{key, gloss}`).
- **Derived at pack** (stripped from the snapshot, recomputed): `h` (per-entry content hash),
  `ipaAmbiguous` (a pure cross-entry function of lemma → set of IPAs), plus the file-level `version`,
  `generatedAt`, `count`.

> Note on `sense`/`senseKey`/`gloss`: these ARE authored (baked in), not derived. They encode
> LLM/human *sense grouping* judgment that cannot be recomputed mechanically. The pack step does **not**
> touch them; the grouping checker (§6) verifies they stay consistent.

## 5. The build: `scripts/seed/pack.mjs`

Replaces `apply-decisions.mjs` on the day-to-day path. Writes the **same output files** as today.

Pseudocode:
```js
const SEED_VERSION = 'sv-2026-06-01'                     // unchanged
const authored = JSON.parse(read('data/seed/sv/wordlist.json'))   // array of entries, no h/ipaAmbiguous

// 1. derive ipaAmbiguous (ported verbatim from apply-decisions.mjs lines ~34-47):
//    per lemma, collect cleanIpa(e.ipa); if a lemma has >1 distinct IPA, mark all its entries.
// 2. canonicalize each entry to the FIXED key order (§5.1), so a hand-edit that reorders keys
//    can never silently change a hash.
// 3. h = sha256(JSON.stringify(entry-without-h)).slice(0,8)   (entryHash in lib/layers.mjs)
// 4. version = SEED_VERSION + '-' + sha256(JSON.stringify(entries-with-h)).slice(0,8)
// 5. seed = { version, generatedAt: new Date().toISOString(), count, entries }
// 6. write minified JSON to  data/seed/sv/seed-sv.json  AND  public/seed-sv.json
//    write { version } to    data/seed/sv/version.json  AND  public/version.json
```

Honor the existing `SEED_OUT_DIR` redirect (see `apply-decisions.mjs`) so the reproducibility test can
pack into a temp dir without clobbering the repo copy.

### 5.1 Fixed key order (critical — read carefully; the naive version is subtly wrong)

`h` and `version` use `JSON.stringify`, which is **key-order sensitive**. `pack` therefore serializes
every entry through one fixed key order (omit absent keys):

```
seedKey, lemma, pos, cefr, gender, ipa, inflections, subDefinitions, altMeanings,
enUncountable, enProper, svUncountable, examples, translation, ipaAmbiguous, sense, h
```
Each `altMeanings[]` object: `key, translation, enUncountable, enProper, examples, senseKey, gloss`.
`sense` object: `{ key, gloss }`.

**But today's shipped seed is NOT in one uniform order.** It has **129 distinct top-level key orders**
(from optional fields appearing or not), and — the trap — in **6 entries** (`innan, nära, tysk,
liberal, hyra, hosta`) `translation` ships *before* `examples`, whereas the order above puts `examples`
first. All 6 are duplicate-collapsed lemmas (2–3 base rows folded into one card — e.g. `nära` adv+adj,
`hosta` verb+noun); in each, `examples` lands *after* `translation`. The exact reason varies per entry
(a sibling's examples folded in by `collapseDuplicates`; a layer-60 example landing on the kept slot,
as for `nära`/`liberal`), so don't rely on a single mechanism. The unifying cause is structural:
`toSeedEntry` is `{ seedKey: kellyId, ...rest }` — nothing imposes a schema order, so today's key order
is emergent from the assembly code paths, not a schema. Canonicalization (Option A) makes that
irrelevant.

There is a genuine trilemma — you cannot have all three:
- **(a)** `version` stays `4898e37e`,
- **(b)** one canonical key order, and
- **(c)** "a human may reorder keys in `wordlist.json` freely."

The committed seed isn't in a single order, so (a)+(b) conflict on those 6 entries.

**Decision — Option A (chosen):** canonicalize every entry to the fixed order above and accept a
**one-time, content-neutral version rehash**. Effect (precomputed and verified against the committed
seed):
- Exactly **6** entries' `h` changes — only their key order differed; every key/value pair is
  identical. **0 entries' decoded content changes.**
- `version` moves once: `sv-2026-06-01-4898e37e` → **`sv-2026-06-01-86b7078c`** (pin this new value).
- On a learner's device, seed-sync sees the new `version`, diffs per-entry `h`, and applies **6
  in-place updates of identical content, 0 adds, 0 deletes** — progress preserved (matched on
  `(seedKey, meaningKey)`).

Option A also *buys* property (c): because `pack` re-serializes through the fixed order, hand-reordering
keys or reflowing whitespace in `wordlist.json` can never change what ships. (Option B — preserve each
entry's on-disk order verbatim to keep `4898e37e` exactly — was rejected: it forfeits (c), the whole
point of a hand-editable snapshot.)

## 6. The checkers (CI-enforced)

CI = the `.github/workflows/deploy.yml` workflow: on every push to `main` it runs `pnpm test`, then
`pnpm build`, then publishes. A failing test blocks the publish. These checks live in `pnpm test`.

| Checker | Reads | Passes when | Status |
|---|---|---|---|
| **Reproducible** | `wordlist.json` → pack → temp; committed `seed-sv.json` | packed `entries` and `version` equal the committed ones (excluding `generatedAt`) | **rewrite** of `seed-reproducible.test.ts` (run `seed:pack`, not `apply-decisions`) |
| **Valid IDs** | `wordlist.json` | every `seedKey` present and unique; every entry's `altMeanings[].key` ≥ 1 and unique within the entry | **new**, trivial |
| **Grouping consistency** | `wordlist.json` + `token-synonyms.json` | for every English phrase produced by ≥2 slots: every producer carries a non-empty group label; a single-label concept has no gloss; a multi-label concept gives every member a non-echo gloss; every promoted+ambiguous slot is covered | **rework** `audit-gloss.mjs` (§6.1) |
| **Per-sense examples** | committed `seed-sv.json` | every split-with-examples word has a sense-specific example for each meaning | **keep** `audit-examples.mjs` unchanged |
| **Reference-list purity** | committed `seed-sv.json` | no `subDefinitions` item bare-repeats the main/promoted meaning | **keep** `audit-subdefs.mjs` unchanged |
| **Token-synonym coverage** | committed `seed-sv.json` + `token-synonyms.json` | every token-ambiguous solo promoted meaning is ruled | **keep** `detect-token-synonyms.mjs` — verify it reads only the shipped seed; repoint if it consults a layer |
| **Seed integrity** | committed `seed-sv.json` | gender homonyms stay distinct; no same-lemma+POS cards share a primary translation | **keep** `seed.test.ts` unchanged |

Because we keep committing `data/seed/sv/seed-sv.json`, every checker that already reads the *shipped*
seed keeps working untouched. Only `audit-gloss` needs rework (it read the sense *layer*).

### 6.1 Reworking `audit-gloss.mjs` to read the snapshot's own labels

Today it treats `layers/50-senses/decisions.json` as the authority for "which slots share a phrase and
how it splits into senses." In the snapshot that authority is the entries' own labels. Change it to:

1. Detect concepts mechanically: `groupConcepts(entries, merges)` from `lib/layers.mjs` (map
   `seedKey`→`kellyId` since it expects `kellyId`; `merges` come from `loadTokenSynonyms`). A concept is
   an English phrase produced by ≥2 slots (primary meaning 0 or a promoted altMeaning).
2. For each concept, read each slot's **group label** (primary → `entry.sense.key`; promoted →
   `altMeanings[k].senseKey`) and **gloss** (primary → `entry.sense.gloss`; promoted →
   `altMeanings[k].gloss`).
3. Group the concept's slots by label; those groups are the senses "as authored."
4. Assert:
   - every slot in a ≥2-producer concept has a **non-empty** label (a blank label = an ungrouped
     member — this is the "`a` newly collided with a group but wasn't grouped" case),
   - a **single-label** concept carries **no** gloss on any slot,
   - a **multi-label** concept carries a **non-echo** gloss on every slot,
   - every promoted + ambiguous slot has a label (coverage).

Keep the `--json` output shape so the test wrapper is a one-line change.

**Caveat — `groupConcepts` is a *lossy* reconstruction of the curated sense partition (read before
tightening assertion 4).** First-token bucketing (`normTr`) is NOT the concept notion the old sense
layer used. The layer could group across first tokens — e.g. concept `of course, you know, after all`
spanned `of course` **and** `you know`/`after all`. When such a concept's other sense loses its only
producer upstream (kellyId 2660, "as you know", was dropped/collapsed), the shipped seed shows just one
first-token bucket ("of course") carrying a label `of course, you know, after all#1`. The reducer keeps
its gloss (the *layer* said 2 senses), so it is **correct, not an invented hint** — but a naïve
"single-label concept ⇒ no gloss" reads it as a regression and (as happened once) blanks 6 legitimate
glosses. Two consequences, both implemented:
- **Single-sense-with-gloss fires only when the label's english equals the concept phrase** (a genuine
  synonym group that shouldn't carry a hint). A label english *broader* than the phrase is a
  cross-first-token fragment → left alone.
- **Blank label is hard-fail + an explicit allowlist** (`BLANK_LABEL_ALLOWLIST`). Some collisions are
  string-identical English for a *genuinely distinct* sense that no mechanical rule can separate from
  "should be grouped" (the `filosofi` / `article (grammar)` line, §4.8) — e.g. `styck` "piece" (the
  counting unit "apiece") vs `bit`/`stycke` "piece". Those are recorded human verdicts, so the check
  stays hard (a *new* ungrouped producer fails CI) without forcing a false merge.

`emptyInMulti` (a genuinely multi-first-token concept missing a gloss) and `echoes` stay 0-tolerance —
those are reliable in the snapshot.

## 7. Editing recipes (what maintenance looks like)

- **Change a translation / example / POS:** edit the field in `wordlist.json`, run `pnpm seed:pack`,
  run `pnpm test`, add a `CHANGELOG.md` line, commit. If the edit touches a `translation`, the grouping
  checker will tell you if you've created or broken a synonym group — fix the labels/glosses of the
  handful of affected words, then commit.
- **Add a word:** append a new entry at the **end** of the array with a fresh unique `seedKey` (do not
  reuse or renumber existing ones). Fill authored fields. Pack + test + commit.
- **Remove a word:** delete its entry. (Seed-sync deletes it from users' devices on next update — this
  is existing, unchanged app behavior.)
- **Never** renumber a `seedKey` or an `altMeanings[].key` — seed-sync matches a learner's progress on
  `(seedKey, meaningKey)`.

## 8. Migration steps (ordered; verify before proceeding)

0. **Branch.** The gate compares against the pre-migration committed `data/seed/sv/seed-sv.json` (git
   already has it). Current `version`: `sv-2026-06-01-4898e37e`.
1. **Write `scripts/seed/pack.mjs`** per §5, including the fixed-key-order canonicalizer.
2. **Generate `data/seed/sv/wordlist.json`** from the current committed `seed-sv.json`: take
   `entries`, strip `h` and `ipaAmbiguous`, pretty-print (2-space indent), **preserve entry order**.
3. **GO/NO-GO GATE (content-neutrality, per §5.1 Option A).** Run `pnpm seed:pack`, then compare the
   packed seed against the pre-migration committed one, decoding both. All three must hold:
   - For **every** `seedKey`, the decoded entry (key-order-insensitive, ignoring `h`) is identical —
     **0 content diffs**. This is the real safety property.
   - Exactly **6** entries' `h` differs (`innan, nära, tysk, liberal, hyra, hosta`), all from the
     `translation`/`examples` reorder — never from a value change.
   - The packed `version` equals the precomputed **`sv-2026-06-01-86b7078c`**.
   Do not proceed until all three hold. (If >6 `h`s change, or any content diff appears: check the §5.1
   key order and the `ipaAmbiguous` derivation — as a diagnostic, keeping `ipaAmbiguous` authored
   instead of derived should not affect the result, since it re-derives to 0 mismatches.)
4. **Rework `audit-gloss.mjs`** (§6.1). Verify `detect-token-synonyms.mjs` reads only the shipped seed.
5. **Update `package.json` scripts:** `seed:pack` = `node scripts/seed/pack.mjs`; make `seed:build` =
   `seed:pack` (drop `seed:compile`/`seed:apply`/`seed:stale` from the everyday path — the batch/
   compile/stale scripts stay in the repo for the archived path). Leave `dev`/`build`/deploy untouched
   (the committed `public/seed-sv.json` means no new build wiring is needed).
6. **Rewrite tests** (§6 + §10). `pnpm test` green.
7. **Archive.** Move `base.json`, `layers.json`, `layers/` → `data/seed/sv/legacy/`. Optionally move
   the layer-only scripts (`compile-layers`, `stale`, `reconcile-stale`, `migrate-ledger`, the
   `batch-*`, `stamp-*` scripts) → `scripts/seed/legacy/`. Keep `lib/layers.mjs` (still used by the
   audits for `groupConcepts`/`normTr`). Keep `token-synonyms.json` where it is. Everything stays in
   git.
8. **Add `data/seed/sv/CHANGELOG.md`** with an initial entry describing this migration.
9. **Docs.** Mark `SEED-PIPELINE-DESIGN.md` as "archived machinery only" and relocate it to
   `data/seed/sv/legacy/`; update `CLAUDE.md`'s "Current Phase", `SPEC.md` §9, and seed references to
   point at the snapshot and this doc.
10. **Acceptance (§8.1). Then stop and ask the user before committing.**

### 8.1 Acceptance criteria

- `pnpm seed:pack` yields `version === sv-2026-06-01-86b7078c`.
- The `seedKey → decoded-content` map is unchanged vs pre-migration (0 content diffs); exactly 6 `h`s
  differ, all from key reordering, none from a value change. **Diff against the frozen `4898e37e`
  content, not the committed `seed-sv.json`** — see the checker-equivalence criterion below for why.
- `pnpm test` is green.
- **The reworked checkers pass clean on ground-truth HEAD.** This is a SEPARATE criterion from "tests
  green," and the one that would have caught the incident below. `pack` a *faithful* HEAD-derived
  `wordlist.json` (strip only `h`/`ipaAmbiguous`), then run `audit-gloss` etc. against it. HEAD is
  correct by construction (the trusted reducer built it), so a reworked checker that flags HEAD is
  wrong and must be fixed (or its exceptions allowlisted with a recorded verdict) — **do not "fix" the
  content to satisfy a new checker.**
- The app boots; for an existing DB on `4898e37e`, seed-sync runs once (version changed) and produces
  **6 in-place content-identical updates, 0 adds, 0 deletes** — progress preserved. Confirm via the
  `/run` or `/verify` skill.

> **Incident note (2026-07-09).** A first pass did the opposite: the reworked `audit-gloss` (naïve
> single-label rule) falsely flagged 6 legit "of course" glosses + `styck`, those were "fixed" in
> `wordlist.json`, and the migration shipped 7 content diffs at `c39d7b85` instead of the neutral
> `86b7078c`. The standing reproducibility test (`pack(wordlist) == committed seed-sv.json`) is **blind
> to this class** — it compares against the already-updated committed file, so a post-gate content edit
> is trivially self-consistent. The content-neutrality gate must diff against the *frozen* pre-migration
> baseline, and "checkers pass on ground-truth HEAD" must be verified explicitly.

## 11. Post-migration capability: targeted LLM re-runs (optional, not required to finish)

The migration does **not** need this — but it is the promised "cheap re-run" path, minus the machinery
that caused pain. The loop is a pair of small scripts:

1. **Extract:** read `wordlist.json`, filter to a subset (e.g. "A1 nouns with no example", "words whose
   English collides with another but have no group label"), project only the fields to work on, write
   task files to `data/scratch/`.
2. **LLM/agent:** run the relevant existing subagent over the task files (they already read/write this
   JSON shape — see the agents in `.claude/agents/`).
3. **Patch:** read the answers, write the touched fields back into `wordlist.json` by `seedKey`.
4. Review the git diff, run `pnpm seed:pack && pnpm test`, add a changelog line, commit.

No ledger, no staleness stamps, no compile, no precedence. The "which words need work" question is a
query you run fresh each time, not state you maintain.

## 12. Gotchas (read before implementing)

- **Hashes are key-order sensitive** — which is *why* `pack` canonicalizes (§5.1). After migration,
  on-disk key order in `wordlist.json` is irrelevant to what ships; the gate checks *decoded content*,
  not bytes. A wrong canonical order surfaces as either a content diff or the wrong pinned version.
- **Do not expect byte-identity — expect content-identity.** Today's seed has 129 key orders and 6
  entries with `translation`/`examples` swapped; canonicalizing rehashes exactly those 6 and moves the
  version to `86b7078c` (§5.1). If a future session frames `4898e37e` as the must-hit target, it will
  wrongly conclude `pack.mjs` is buggy — the target is `86b7078c` with 0 content diffs.
- **`generatedAt` is non-deterministic** — never include it in any comparison (compare `entries` +
  `version`, not raw bytes).
- **Never reorder/renumber entries, `seedKey`s, or `meaningKey`s.** Append new words at the end.
- **Group labels (`english#N`) are opaque strings.** What matters is that grouped slots share one; when
  a concept splits into senses, give distinct labels and a gloss per sense.
- **`wordlist.json` is ~5 MB pretty (~125k lines).** Fine for git; diffs stay localized. Some editors
  are slow to open it — that is a UX note, not a correctness issue. If it becomes unwieldy, sharding by
  CEFR is a future option (out of scope here).
- **Do not delete the `legacy/` tree.** It is the only home of the raw LLM ledgers and the sole re-pass
  path.
- **The reproducibility test does NOT guard content-neutrality.** `seed-reproducible.test.ts` compares
  `pack(wordlist)` against the *committed* `seed-sv.json` — so once you've re-packed, any content edit
  is trivially self-consistent and passes. Content-neutrality is a one-time migration property: diff
  against the *frozen* `4898e37e` baseline (§8.1). A green suite alone did not catch a 7-entry content
  drift; only the frozen-baseline diff + "checkers pass on ground-truth HEAD" did.
- **`groupConcepts` (first-token) is a lossy view of the curated sense partition** — see the §6.1
  caveat before treating a single-first-token bucket as "single-sense." A gloss whose label english is
  broader than the phrase is legitimate; blanking it is a regression.
